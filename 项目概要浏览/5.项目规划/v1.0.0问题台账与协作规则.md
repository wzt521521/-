## 3. Issue 规则

1. 一个 Issue 只对应一个任务 ID；不得把多个 ID 合并为无法独立验收的大任务。
2. Issue 必须包含范围、验收标准、验证命令、依赖和风险；P0 必须说明失败与回滚路径。
3. 状态变化、阻断原因和验收证据记录在 Issue，不以聊天记录替代。
4. 只有代码/文档经 PR 合入发行分支、自动化测试或可重复脚本通过、配置和回滚说明同步后才能关闭。
5. P2 移入 `v1.1` 必须保留 Issue 和决策记录，不得直接删除任务。

## 4. 分支与 PR 规则

- 从最新 `release/1.0.0-integration` 创建短生命周期分支。
- 分支格式：`fix/<task-id>-<topic>`、`feat/<task-id>-<topic>`、`docs/<task-id>-<topic>` 或 `chore/<task-id>-<topic>`。
- PR 标题格式：`[TASK-ID] type: summary`，正文使用仓库 PR 模板并通过 `Closes #<issue>` 关联 Issue。
- 一个 PR 只解决一个任务 ID；共享的机械修改应先拆分依赖 Issue，不得再次直接推入主线。
- PR 必须通过 `backend`、`frontend`、`data-pipeline`、`compose`，至少 1 人批准，最后推送者不能自批，所有审查会话必须解决。
- 合并顺序由依赖关系和 P0/P1 优先级决定；禁止因“改动很小”绕过门禁。

## 5. 完成定义

- 变更已通过 PR 合入发行分支。
- 自动化测试或可重复验收脚本覆盖成功、失败和权限边界。
- 在干净环境可复现，不依赖开发者机器残留状态。
- 文档、环境变量、数据库迁移、部署和回滚配置同步更新。
- Issue 中记录验证命令、关键结果、合入 PR 和残余风险。

- 全部发行任务：[Milestone `v1.0.0`](https://github.com/wzt521521/Career-Ability-Big-Data-Platform/milestone/1)
